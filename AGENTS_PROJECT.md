# AGENTS_PROJECT.md

このファイルは、そのプロジェクト固有の AI agent 向け運用メモです。
共通ルールは dotfiles の `templates/AGENTS_BASE.md` に集約します。

## Goal
- 放射線治療装置の精度管理のためのwindton lutz テストの自動解析用ソフト。
- 現状wanalyzer2.0として開発したものの、操作の柔軟性に難があるため、GUIベースにした上で各設定等を柔軟にユーザーが変更できるようにしたい。
- 現在、csvをベースにデータの管理を行っているため、データベースはsqliteを使用し、必要テーブルのクエリなどをできるようにする。

## Context
- 解析する画像は放射線治療装置からdicomでは出力できず、bmpでの出力になる。そのため、dicomのタグなどを利用した解析ができない。一方で自身の触れていない総力は出力できる可能性があるため、画像の読み込み自体はpng, bmp, dicom, tiffには対応させるべき。
- 使用するOSはwindowsをベースにするが、pythonで実装するため、クロスプラットフォームで良い。
- tkinterを使用して一部GUIを実装しているが、特にこだわりはないため、pyside6を使用するように変更予定。
- 解析内容としては、機械のレーザー位置を真の中心とし、それに合わせて設置した小球体を矩形の照射野で撮影する。その画像を各方向(ガントリ、コリメータ、テーブルの角度変化)から撮影し、レーザー位置と照射野の中心位置のずれを解析する。解析方法は取得画像から球体と矩形照射野のそれぞれの中心位置を割り出し、その二次元的な位置ズレを算出し、方向と実距離を出して記録しているだけである。
- 現在は撮影を行う順序を固定しており、それに対してハードコーディングで対応している。デイリーとしての解析はこれで済んでいるが、点検が必要な時にも一部撮影条件のみでwindton lutzテストを行うこともあるため、日常使用と一時的な使用でデータの蓄積は事実上同じデータとして連続するが、その時々で必要な撮影条件が異なってしまうことがある。これに関してはガントリ、コリメータ、テーブルの各パラメータを逐一入れる必要がある時と、手間を減らして同じ条件で連続的に解析するモードなどで対応する必要があるが、ワークフローを検討中である。案の一つとしては、事前に画像の順番と撮影条件を定義した解析条件リストを用意し、一連の解析画像を読み込んだ後にリストから解析条件リストからどの条件で解析するかを選択して解析する。データは各条件でそれぞれデータベースに入れ、必要に応じてクエリして結果を取り出せるようにする。また、解析時にもこれが日常点検か一時的な点検かを記録し、それぞれのデータとして合わせて表示したり、フィルタリングしたりできるようにする。というのを考えている。
- 日々の点検用の画像はsample/setに入っている画像で、画像の順番はガントリ180.1, 270, 0, 90, 179.9, コリメータ270, 0, 90, 180, カウチ回転90, 45, 0, 315, 270の順で14枚取得している。これは実際の運用でもこの順番で撮影しているが、別の施設ではまたやり方が異なる場合があるため、柔軟性を上げ、先ほどの条件リストで管理したい。
- sample/sizeの画像は、照射野の大きさを変えて撮影した画像で、解析パラメータのテストを行う際に使用している。
- test実行でできたファイル、ディレクトリ、生成物に関しては確認なしに削除して良い。
